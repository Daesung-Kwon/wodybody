# BurnFat V2 서비스 확장 기획

## 1. 기획 개요

| 항목 | 내용 |
|------|------|
| **목표** | MVP 기반 빠르게 확장 가능한 서비스 추가 |
| **원칙** | 기존 데이터·인프라 활용, 점진적 확장 |
| **기술 제약** | Supabase + Cloudflare Pages (Functions 없이 SPA 우선) |
| **UI/UX 기본** | **모바일 화면 최적화 기본** – 지인 대결은 모바일 접속 비중이 높음 |

## 1.1 서비스 철학: 신뢰 기반 공개형

| 항목 | 내용 |
|------|------|
| **데이터 입력** | 대결방 접속자 **누구나** 특정 참가자(닉네임)의 데이터 입력 가능 |
| **데이터 공개** | 모든 참가자의 데이터가 **전원에게 공개**, 함께 모니터링 |
| **신뢰** | 지인·동료 기반 대결, 서로의 기록을 투명하게 확인·보완 |

**확장 시 준수**
- "내 데이터만" 접근 제한 없음 → **공동 입력·공동 확인** UX
- RLS/권한: 대결방 코드만 알면 읽기·쓰기 동일 (기존 MVP 유지)

## 2. 확장 서비스 제안

### 2.1 주별 바디 데이터 수집 + AI 조언 (우선 검토)

**개념**
- 대결 기간 중 **주 1회** 추가 데이터 입력 (나이, 몸무게, 성별, 체지방률 등)
- 종료일 전까지 누적 데이터를 활용해 **AI 기반 체지방 감량 조언** 제공

**수집 데이터 (예시)**

| 필드 | 타입 | 설명 |
|------|------|------|
| age | int | 나이 |
| gender | enum | 성별 (M/F) |
| weight_kg | decimal | 몸무게, kg |
| height_cm | decimal | 키, cm (선택) |
| body_fat_rate | decimal | 체지방률 |

**기술 구현**
- **DB**: `weekly_logs` 테이블 (participant_id, week_no, recorded_at, ...)
- **AI**: Supabase Edge Functions + OpenAI API 또는 Cloudflare Workers + AI
- **프론트**: 대결방 내 "주간 기록" 탭 추가

**가치**
- 단순 감량 비교 → **맞춤형 조언**으로 전환
- 체지방률 변화 추이 + 개인 정보로 현실적인 목표·전략 제시

---

### 2.2 목표 체지방률 설정 + 진행률 시각화

**개념**
- 참가 시 **목표 체지방률** 입력
- 주간 기록·진행률을 차트로 표시

**수집 데이터**
- `participants`에 `target_body_fat` 컬럼 추가
- `weekly_logs`와 연계해 진행률 계산

**가치**
- 목표 대비 진행 상황을 한눈에 파악
- 동기 부여 및 이탈 감소

---

### 2.3 대결 결과 리포트 (PDF/이미지)

**개념**
- 대결 종료 후 **개인 결과 리포트** 생성
- 체지방 변화, 순위, 간단한 통계 포함

**기술**
- 클라이언트: html2canvas + jsPDF 또는 react-pdf
- 또는 Edge Function에서 PDF 생성 후 Storage 저장

**가치**
- SNS 공유, 기록 보관
- 서비스 브랜딩 강화

---

### 2.4 리마인더 / 알림

**개념**
- 주간 기록 입력, 종료일 인증 등 **알림 발송**

**기술**
- Supabase Database Webhooks → 외부 서비스 (Resend, SendGrid 등)
- 또는 Cloudflare Workers Cron + 이메일 API

**전제**
- 참가 시 이메일 또는 푸시 수집 필요

---

### 2.5 대결 템플릿 / 프리셋

**개념**
- "4주 챌린지", "8주 챌린지" 등 **기간·참가비 프리셋**
- 대결 생성 시 원클릭 선택

**가치**
- 생성 UX 단순화
- 반복 사용자 증가

---

### 2.6 팀/그룹 대결

**개념**
- 개인 대결 → **팀 vs 팀** 대결
- 팀별 평균 감량 또는 합산으로 순위

**기술**
- `challenges`에 `mode` (solo/team) 추가
- `teams` 테이블, `participants.team_id` FK

## 4. 1순위 상세: 주별 데이터 + AI 조언

### 4.1 데이터 모델 (제안)

```sql
-- 주간 로그 (participant별 주차별 1회)
CREATE TABLE weekly_logs (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  participant_id UUID NOT NULL REFERENCES participants(id) ON DELETE CASCADE,
  week_no INTEGER NOT NULL,                    -- 1, 2, 3, 4...
  recorded_at DATE NOT NULL,                   -- 기록일
  age INTEGER,
  gender TEXT CHECK (gender IN ('M', 'F')),
  weight_kg DECIMAL(5,2),
  height_cm DECIMAL(5,1),
  body_fat_rate DECIMAL(5,2),
  note TEXT,                                   -- 자유 메모
  created_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(participant_id, week_no)
);

-- participants 확장 (참가 시 1회 입력)
ALTER TABLE participants ADD COLUMN IF NOT EXISTS age INTEGER;
ALTER TABLE participants ADD COLUMN IF NOT EXISTS gender TEXT;
ALTER TABLE participants ADD COLUMN IF NOT EXISTS height_cm DECIMAL(5,1);
ALTER TABLE participants ADD COLUMN IF NOT EXISTS target_body_fat DECIMAL(5,2);
```

### 4.2 AI 조언 시나리오

| 입력 | AI 출력 예시 |
|------|--------------|
| 30대 M, 1주차 -0.5%p | "1주차에 0.5%p 감소는 양호합니다. 주 0.3~0.5%p가 무리 없는 범위입니다." |
| 40대 F, 2주차 +0.2%p | "2주차 소폭 상승은 수분·식사 타이밍 영향일 수 있습니다. 단백질 섭취와 수면을 점검해보세요." |
| 3주 연속 정체 | "정체 구간입니다. 운동 강도 조절이나 식단 점검이 필요할 수 있습니다." |

### 4.3 구현 단계

| 단계 | 내용 | 예상 공수 |
|------|------|----------|
| 1 | `weekly_logs` 테이블 + 참가 시 기본 정보 입력 UI | 1~2일 |
| 2 | 주간 기록 입력 폼 + 목록/차트 | 1~2일 |
| 3 | Supabase Edge Function + OpenAI 연동 | 1~2일 |
| 4 | AI 조언 노출 UI (모달/카드) | 0.5일 |

## 4.4 모바일 최적화 · UI/UX 기본 원칙

**전제**: 모든 화면은 **모바일 우선(Mobile-first)** 으로 설계. 데스크톱은 확장 레이아웃.

### 4.4.1 참가하기 플로우 (모바일 최적화)

| 단계 | UI/UX | 설명 |
|------|-------|------|
| 1 | **참가하기 영역** | 닉네임 입력 + 참가 버튼만 노출. 영역 최소화 |
| 2 | **참가 등록** | 닉네임으로 participants INSERT |
| 3 | **기본정보 팝업** | 등록 성공 시 **새 팝업(모달)** 에서 age, gender, height_cm, target_body_fat 입력 |
| 4 | **저장** | 팝업에서 participants UPDATE 후 닫기 |

**이유**: 참가하기 영역에 기본정보 입력란이 모두 있으면 모바일에서 영역 차지가 과도함. 닉네임만 받고 등록 후 팝업으로 분리하면 첫 화면이 간결해짐.

### 4.4.2 모바일 UX 원칙

| 원칙 | 적용 |
|------|------|
| **터치 타겟** | 버튼·입력 영역 최소 44px 높이 |
| **스크롤** | 한 화면에 과도한 폼 배치 지양, 스텝·팝업으로 분리 |
| **입력 폼** | 모바일 키보드 타입 적절히 지정 (number, tel 등) |
| **모달** | 하단 시트(Bottom Sheet) 또는 전체 화면 모달로 터치 친화적 |

## 5. 기술 고려사항

### 5.1 AI 연동 옵션

| 방식 | 장점 | 단점 |
|------|------|------|
| **Supabase Edge Function** | Supabase 내 통합, DB 직접 접근 | Deno 런타임 |
| **Cloudflare Workers + AI** | 엣지 실행, Workers AI 가능 | 별도 배포 |
| **클라이언트 직접 호출** | 구현 단순 | API 키 노출 위험, 비권장 |

**권장**: Supabase Edge Function에서 OpenAI API 호출. (공개형: 대결방 코드 기반 접근만 검증, participant 소유 제한 없음)

### 5.2 비용

- OpenAI API: 토큰당 과금 (소규모 사용 시 저비용)
- Supabase: 무료 플랜 내 Edge Function 호출 제한 확인
- Cloudflare Pages: 무료

## 6. 공개형·신뢰 기반 추가 기획

### 6.1 공개형에 맞는 UX/카피

| 구분 | 제안 |
|------|------|
| **입력 버튼** | "OO님 주간 기록 입력" (본인/타인 구분 없이 동일) |
| **안내 문구** | "대결방 참가자 누구나 서로의 기록을 입력·확인할 수 있습니다" |
| **탭/섹션** | "주간 기록" (개인 락 없음, 전원 데이터 나열) |

### 6.2 공동 모니터링 강화 기능

| 기능 | 설명 | 가치 |
|------|------|------|
| **참가자별 기록 현황** | "이번 주 N명 기록 완료, M명 미기록" 한눈에 표시 | 누가 빠졌는지 확인 → 친구가 대신 입력 유도 |
| **전체 추이 한눈에** | 참가자 전원의 주간 체지방 변화를 한 화면에 차트 | 함께 모니터링, 서로 격려·경쟁 |
| **최종 수정 시각** | 각 기록에 "N일 전 입력" 표시 | 투명성, 최신 여부 확인 |

### 6.3 신뢰 기반 활용

| 기능 | 설명 |
|------|------|
| **대결방 규칙/약속** | (선택) "우리 대결방 규칙" 텍스트 영역 – 신뢰·공정성 약속 |
| **공동 케어** | 미기록자에게 "이번 주 기록 부탁해요" 등 간단 안내 노출 |
| **AI 조언 공개 범위** | 참가자별 AI 조언도 전원 공개 (선택) – 함께 조언 공유 |

### 6.4 악용 방지 (최소 수준)

| 위험 | 대응 |
|------|------|
| 대결방 코드 유출 | 코드 공유 범위는 생성자·참가자 책임 (현 구조 유지) |
| 허위 데이터 입력 | 신뢰 기반: 지인 대결 전제, 악의적 입력은 참가자 간 협의 |
| (선택) 수정 이력 | `updated_at`만 유지, "누가" 수정은 추적하지 않음 (익명성 유지) |
